Вариант 2. Профессиональный вариант (Идеальное решение)
Поскольку ты уже знаком с nftables по первому серверу, использовать socat на сервере-прокладке для UDP (и даже для TCP) — это, честно говоря, стрельба из пушки по воробьям.

Зачем гонять трафик через тяжелую программу в пространстве пользователя (socat), если сам сервер на уровне ядра Linux может пересылать эти пакеты со скоростью света вообще со 0% нагрузки на процессор?

Ты можешь полностью выключить обе службы socat и написать одно короткое правило в nftables на сервере-прокладке, которое объединит и TCP, и UDP.

Как это сделать через nftables на прокладке:
Установи nftables на сервере-прокладке (если еще не установлен):

Bash
apt install nftables -y
Открой его конфиг /etc/nftables.conf:

Bash
nano /etc/nftables.conf
Очисти всё и вставь этот конфиг транзита (DNAT):

Фрагмент кода
#!/usr/sbin/nft -f

flush ruleset

table ip nat {
    chain prerouting {
        type nat hook prerouting priority dstnat; policy accept;

        # Всё, что прилетает на порт 40000 (и TCP, и UDP), 
        # ядро Linux мгновенно и без лишних процессов перенаправляет на целевой VPN
        tcp dport 40000 dnat to 38.244.202.45:443
        udp dport 40000 dnat to 38.244.202.45:443
    }

    chain postrouting {
        type nat hook postrouting priority srcnat; policy accept;
        
        # Подменяем IP сервера-прокладки, чтобы основной VPN-сервер знал, куда вернуть ответ
        masquerade
    }
}
Включи и запусти службу фаервола:

Bash
systemctl enable nftables --now
В чем магия этого решения?
И TCP, и UDP работают внутри одного файла конфигурации.

Нагрузка на процессор = 0%. Пакеты не перекладываются из одной программы в другую, маршрутизацией занимается само ядро Linux.

Никаких вылетов и лимитов. Понятие Too many open files или лимит процессов fork здесь вообще перестают существовать, так как файлы для транзитных пакетов не открываются. По сути, твоя прокладка превращается в чистокровный аппаратный роутер.


Плодить кучу служб socat под каждый порт, да еще и разделять их на TCP и UDP — это огромная нагрузка на систему и кошмар для администрирования. Если у тебя будет, например, 5 серверов, тебе придётся держать 10 запущенных процессов socat.

В nftables всё это прописывается в одном-единственном файле конфигурации. Ядро Linux будет само распределять порты по нужным серверам.

Вот как грамотно и красиво настроить этот процесс.

Шаг 1. Полностью останавливаем и отключаем socat
Поскольку nftables будет перехватывать пакеты прямо «на лету» на уровне ядра, службы socat нам больше вообще не нужны. Они будут только зря занимать порты и тратить оперативную память.

Выключи их:

Bash
systemctl stop socat socat-udp 2>/dev/null
systemctl disable socat socat-udp 2>/dev/null
Шаг 2. Пишем единый конфиг для nftables
Открой файл конфигурации фаервола:

Bash
nano /etc/nftables.conf
Очисти всё, что там есть, и вставь следующую структуру. Ты можешь добавлять сюда сколько угодно серверов, просто дописывая по две строчки (для TCP и UDP):

Фрагмент кода
#!/usr/sbin/nft -f

# Очищаем старые правила, чтобы не было конфликтов
flush ruleset

table ip nat {
    chain prerouting {
        type nat hook prerouting priority dstnat; policy accept;

        # =========================================================
        # НАПРАВЛЕНИЕ 1: Первый VPN-сервер (например, твоя Германия)
        # Входной порт: 40000 -> пересылаем на порт 443 целевого сервера
        # =========================================================
        tcp dport 40000 dnat to 38.244.202.45:443
        udp dport 40000 dnat to 38.244.202.45:443

        # =========================================================
        # НАПРАВЛЕНИЕ 2: Второй VPN-сервер (например, Нидерланды)
        # Входной порт: 40001 -> пересылаем на порт 443 второго сервера
        # =========================================================
        tcp dport 40001 dnat to 45.67.89.10:443
        udp dport 40001 dnat to 45.67.89.10:443

        # =========================================================
        # НАПРАВЛЕНИЕ 3: Третий VPN-сервер (если появится в будущем)
        # Входной порт: 40002 -> пересылаем на порт 443 третьего сервера
        # =========================================================
        # tcp dport 40002 dnat to 192.168.1.50:443
        # udp dport 40002 dnat to 192.168.1.50:443
    }

    chain postrouting {
        type nat hook postrouting priority srcnat; policy accept;
        
        # Маскарадинг обязателен — он подменяет IP-адрес отправителя на IP прокладки,
        # чтобы конечные VPN-серверы знали, куда отправлять ответный трафик.
        masquerade
    }
}
(Замени 45.67.89.10 на реальный IP-адрес твоего второго VPN-сервера).

Шаг 3. Включаем и перезапускаем службу
Чтобы ядро Linux применило эти правила маршрутизации, выполни:

Bash
# Разрешаем запуск nftables при старте системы
systemctl enable nftables

# Перезапускаем службу для применения новых правил
systemctl restart nftables
Проверить, что правила успешно загрузились в память, можно командой:

Bash
nft list ruleset
Ты увидишь красивую и понятную таблицу со всеми своими портами и адресами назначения.

Почему эта схема идеальна?
Никакой путаницы со службами: У тебя на сервере-прокладке работает ровно одна системная служба — nftables.

И TCP, и UDP из коробки: Больше не нужно переживать, что UDP «повесит» сервер. Для ядра Linux нет разницы, оно обрабатывает миллиарды таких пакетов в секунду без создания процессов-зомби.

Легко масштабировать: Если завтра у тебя появится 4-й, 5-й или 10-й VPN-сервер, тебе больше не придётся создавать новые файлы служб в systemd. Ты просто зайдёшь в /etc/nftables.conf, допишешь по аналогии две строчки с новым портом и IP, сделаешь systemctl restart nftables, и всё сразу заработает!
